# 提示词模板系统设计

基于 OpenCode 项目，梳理终端 AI 助手的提示词工程架构与实现策略。

---

## 1. 设计目标

提示词模板系统的核心目标是：**为不同场景、不同模型，动态组装最合适的系统提示词**，让 AI 在每次对话开始时都具备正确的角色认知、环境感知和项目背景。

---

## 2. 核心设计原则

### 2.1 角色分离

不同场景对 AI 的要求截然不同，不能共用一份提示词：

- **Coder**：需要详细的编码规范、工具使用策略、代码风格约束
- **Task**：需要简洁直接，适合快速问答和简单操作
- **Title**：只需要一句话概括能力，生成简短标题
- **Summarizer**：需要提取和保留关键上下文信息

### 2.2 提供商适配

不同 LLM 对提示词的"脾性"不同：

- Claude（Anthropic）对 XML 标签、结构化格式更友好
- GPT（OpenAI）对函数调用描述方式更敏感
- 本地模型可能对指令遵循能力较弱，需要更明确的约束

### 2.3 上下文动态注入

静态提示词无法满足动态环境需求，需要实时注入：

- 当前工作目录、Git 状态
- 操作系统平台、当前日期
- 项目特定规范文件（OpenCode.md 等）

---

## 3. 整体架构

```
┌────────────────────────────────────────────┐
│           统一入口 GetAgentPrompt            │
│  输入：AgentName + ModelProvider             │
│  输出：组装完成的系统提示词                   │
├────────────────────────────────────────────┤
│              第一层：角色选择                  │
│  根据 AgentName 选择基础模板                  │
│  Coder / Task / Title / Summarizer          │
├────────────────────────────────────────────┤
│              第二层：提供商适配                │
│  根据 ModelProvider 微调基础模板              │
│  OpenAI 风格 / Anthropic 风格 / 通用风格      │
├────────────────────────────────────────────┤
│              第三层：环境信息注入              │
│  动态生成：工作目录、Git状态、平台、日期        │
│  项目文件列表（ls 当前目录）                   │
├────────────────────────────────────────────┤
│              第四层：项目上下文加载            │
│  从 contextPaths 读取项目特定文件             │
│  OpenCode.md、.cursorrules 等               │
└────────────────────────────────────────────┘
```

---

## 4. 四层组装机制详解

### 4.1 第一层：角色选择

系统预定义四种 Agent 角色，每种有独立的基础提示词：

**Coder 角色**

最复杂的提示词，包含：
- 身份定义（OpenCode CLI 编码助手）
- 能力说明（可调用工具、执行命令、修改文件）
- 编码规范（不添加注释、保持代码风格一致、使用绝对路径）
- 工作流指导（先理解再修改、运行测试验证、不擅自提交代码）
- 沟通风格（简洁、直接、终端友好格式）
- 工具使用策略（优先搜索现有代码、并行调用独立工具）
- OpenCode.md 记忆机制说明

**Task 角色**

简化版提示词，强调：
- 简洁回答，避免冗长
- 直接回应用户问题
- 返回绝对路径

**Title 角色**

极简提示词，约束：
- 不超过 50 字符
- 单行，无标点符号
- 概括用户首条消息

**Summarizer 角色**

专注信息提取，要求保留：
- 已完成的工作
- 当前进行中的任务
- 正在修改的文件
- 下一步计划

### 4.2 第二层：提供商适配

以 Coder 角色为例，针对 OpenAI 和 Anthropic 提供不同版本的基础模板：

**OpenAI 版本特点**
- 更强调"Agent"概念（持续工作直到任务完成）
- 明确提及 function calling 机制
- 更详细的编码准则列表
- 强调沙箱环境和 git 工作流

**Anthropic 版本特点**
- 更强调"Memory"机制（OpenCode.md）
- 包含大量具体示例（展示合适的回答长度）
- 更详细的沟通风格约束（避免前奏/后奏）
- 强调工具使用策略（Agent 工具、并发调用）

**设计意图**：不同模型训练数据不同，对提示词结构的敏感度不同，针对性优化能提升输出稳定性。

### 4.3 第三层：环境信息注入

每次对话前动态收集并注入环境上下文：

```
<env>
Working directory: /path/to/project
Is directory a git repo: Yes
Platform: linux
Today's date: 2026/4/28
</env>
<project>
[当前目录文件列表]
</project>
```

**环境信息包含**

- 当前工作目录绝对路径
- 是否为 Git 仓库
- 操作系统平台
- 当前日期
- 项目根目录文件列表（通过 ls 工具获取）

**设计意图**：让 AI 知道自己在什么环境下运行，避免使用相对路径、避免询问"你在什么目录"。

### 4.4 第四层：项目上下文加载

从配置文件指定的路径读取项目特定指令文件，自动追加到提示词：

**默认加载的文件列表**

```
.github/copilot-instructions.md
.cursorrules
.cursor/rules/
CLAUDE.md
CLAUDE.local.md
opencode.md
opencode.local.md
OpenCode.md
OpenCode.local.md
OPENCODE.md
OPENCODE.local.md
```

**加载逻辑**

- 使用 sync.Once 缓存，进程内只加载一次
- goroutine 并发读取多个文件，减少启动延迟
- 支持目录递归读取（以 / 结尾的路径）
- 自动去重（不区分大小写）
- 文件不存在则静默跳过

**内容格式**

每个文件内容会被包装为：
```
# From: /path/to/file
[文件原始内容]
```

**设计意图**：兼容现有生态（Cursor、Copilot、Claude），同时提供项目级记忆能力。

## 5. OpenCode.md 项目记忆机制

这是提示词系统中最具特色的设计之一。

### 5.1 定位

项目根目录下的 Markdown 文件，作为**跨会话持久化的项目记忆**。它解决了一个核心问题：**每个会话独立，AI 每次都需要重新"学习"项目**。

### 5.2 内容建议

- 常用命令（build、test、lint、typecheck）
- 代码风格偏好（命名规范、导入风格、错误处理）
- 项目结构说明（重要目录、架构约定）
- 技术栈信息（框架版本、特殊配置）

### 5.3 使用方式

**自动加载**

系统启动时自动检测并读取 OpenCode.md，注入到 Coder 和 Task Agent 的系统提示词。

**渐进式积累**

Coder 提示词中明确告知 AI：

> "当你花时间搜索 typecheck、lint、build 或 test 命令时，应该询问用户是否可以添加到 OpenCode.md。同样，当学习到代码风格偏好或重要代码库信息时，询问是否可以添加到 OpenCode.md 以便下次记住。"

**设计意图**：不是 AI 擅自修改，而是**征得用户同意后更新**，避免污染记忆文件。

### 5.4 与会话摘要的区别

| | OpenCode.md | 会话摘要 |
|--|-------------|----------|
| 内容 | 项目通用知识 | 某次对话的具体内容 |
| 生命周期 | 持久化文件，随项目存在 | 随会话存在 |
| 使用方式 | 自动注入所有会话 | 只在原会话内使用 |
| 维护者 | 用户 + AI（经同意） | 系统自动生成 |
| 目的 | 避免重复说明项目背景 | 解决长对话上下文超限 |

## 6. 性能优化

### 6.1 缓存机制

项目上下文文件通过 `sync.Once` 实现进程级缓存：

- 首次调用时并发读取所有文件
- 后续调用直接返回缓存内容
- 避免每次对话都重复读取磁盘

### 6.2 并发加载

多个上下文文件通过 goroutine 并行读取：

- 每个路径一个 goroutine
- 使用 channel 收集结果
- 使用 WaitGroup 等待全部完成
- 使用互斥锁保护去重映射

### 6.3 懒加载

环境信息（如当前目录文件列表）在需要时才生成，而非启动时预计算。

## 7. 提示词工程策略

### 7.1 具体化优于抽象化

提示词中包含大量具体示例，而非抽象描述：

```
<example>
user: 2 + 2
assistant: 4
</example>

<example>
user: what is 2+2?
assistant: 4
</example>
```

**原因**：LLM 对示例的学习效果远好于规则描述。

### 7.2 约束分层

将约束按重要性分层：

- **硬性约束**（ NEVER 开头）：不添加注释、不提交代码、不猜测路径
- **软性约束**（应该/建议）：询问是否更新 OpenCode.md、解释命令用途
- **风格约束**（简洁/直接）：回答长度、格式要求

### 7.3 上下文窗口管理

提示词设计时考虑 token 消耗：

- 基础提示词控制在合理长度（Coder 约 2-3K tokens）
- 项目上下文可选加载（用户可配置 contextPaths）
- 环境信息精简（只列文件名，不列内容）
- 避免冗余重复（如多次强调同一规则）

## 8. 扩展性设计

### 8.1 新增角色

如需新增 Agent 角色：

1. 在 AgentName 枚举中新增角色名
2. 在 GetAgentPrompt 的 switch 中新增 case
3. 创建对应的提示词生成函数
4. 可选：在 config 中配置该角色使用的模型

### 8.2 新增提供商

如需支持新的 LLM 提供商：

1. 在 ModelProvider 枚举中新增提供商
2. 为该提供商创建基础提示词变体（如果需要）
3. 在 Provider 层实现统一的 SendMessages/StreamResponse 接口

### 8.3 自定义上下文

用户可通过配置文件扩展上下文路径：

```json
{
  "contextPaths": [
    "README.md",
    "docs/",
    "my-project-rules.txt"
  ]
}
```

## 9. 设计亮点总结

| 设计点 | 说明 |
|--------|------|
| **角色分离** | 不同场景使用不同提示词，避免一份提示词既要又要 |
| **提供商适配** | 针对不同 LLM 微调提示词风格，提升输出稳定性 |
| **四层组装** | 基础模板 + 提供商适配 + 环境信息 + 项目上下文，灵活组合 |
| **项目记忆** | OpenCode.md 实现跨会话项目知识共享 |
| **并发加载** | goroutine 并行读取上下文文件，优化启动性能 |
| **缓存优化** | sync.Once 确保进程内只加载一次项目上下文 |
| **生态兼容** | 同时支持 Cursor、Copilot、Claude 的规则文件 |
| **示例驱动** | 用具体示例而非抽象规则引导模型行为 |

---

## 10. 一句话概括

OpenCode 的提示词系统 = **"一个按角色分离、按提供商适配、动态注入环境和项目上下文的四层提示词组装引擎"**。

它让 AI 从"通用助手"变成了"懂项目背景、知环境状态、按场景行事的专职助理"。

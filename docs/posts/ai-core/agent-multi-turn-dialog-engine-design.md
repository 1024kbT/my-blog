# Agent 多轮对话引擎设计

基于 OpenCode 项目，梳理终端 AI 助手的核心对话引擎设计思路与关键机制。

---

## 1. 整体架构：异步事件驱动

Agent 引擎采用 **"异步执行 + 事件流输出"** 的架构：

```
用户输入
   ↓
Run(sessionID, content) 
   ↓
启动 goroutine 异步执行对话流程
   ↓
通过 channel 返回 AgentEvent（流式事件）
   ↓
上层（TUI 或非交互模式）消费事件并展示
```

**设计意图**

- 不阻塞主线程，UI 保持响应
- 支持流式输出，用户体验好
- 便于集成取消机制

---

## 2. 核心对话循环：Tool Use Loop

这是 Agent 引擎最核心的设计。对话不是一次请求-响应就结束，而是可能经历 **多轮"思考→工具调用→执行→再思考"** 的循环。

### 2.1 流程图

```
开始
  │
  ▼
构造消息历史 [系统提示词 + 历史消息 + 用户新消息]
  │
  ▼
发送给 LLM（StreamResponse）
  │
  ▼
实时接收事件流 ──→ 内容增量（展示给用户）
  │                思考过程（Reasoning）
  │                工具调用请求
  ▼
AI 完成响应？
  │
  ├─ 是 → 返回最终答案，结束
  │
  └─ 否（请求调用工具）
          │
          ▼
    解析工具调用（可能多个并行）
          │
          ▼
    权限检查（敏感工具需要确认）
          │
          ▼
    执行工具（bash、edit、view 等）
          │
          ▼
    构造工具结果消息
          │
          ▼
    将工具结果追加到消息历史
          │
          └──────→ 回到"发送给 LLM"（继续循环）
```

### 2.2 为什么需要循环？

AI 的推理和工具调用是交织的：

1. AI 收到任务 → 分析需要做什么
2. AI 决定查看文件 → 调用 `view` 工具
3. 拿到文件内容 → 继续思考
4. AI 决定修改代码 → 调用 `edit` 工具
5. 修改完成 → 验证结果 → 调用 `bash` 运行测试
6. 测试通过 → 给出最终答案

### 2.3 循环终止条件

- AI 的 `finish_reason` 不是 `tool_use`（即不再需要调用工具）
- 用户取消请求
- 发生错误

## 3. 消息历史管理

### 3.1 消息角色设计

对话历史包含四种角色的消息：

| 角色 | 说明 |
|------|------|
| **System** | 系统提示词（注入到第一条或作为独立消息） |
| **User** | 用户输入（文本 + 附件） |
| **Assistant** | AI 回复（文本 + 工具调用请求） |
| **Tool** | 工具执行结果（多个工具结果合并为一条） |

### 3.2 摘要截断机制

当会话存在 `SummaryMessageID` 时，加载历史消息会智能截断：

```
原始历史：[Msg1] [Msg2] [Msg3] [Msg4] [Summary] [Msg5] [Msg6]
                                    ↑
                              截断点
加载后：[Summary] [Msg5] [Msg6] [新用户消息]
```

**设计巧妙之处**

- 摘要消息被保留在历史中（作为 Assistant 角色）
- 它之前的原始消息被丢弃，节省 token
- 它之后的正常对话继续保留
- 对 AI 来说，摘要就像"之前的同事交接给我的备忘录"

## 4. 流式响应处理

OpenCode 与 LLM 的交互是**流式**的（SSE/Streaming），不是等全部生成完再处理。

### 4.1 事件类型

从 LLM 返回的流被解析为结构化事件：

| 事件类型 | 触发时机 | 处理动作 |
|----------|----------|----------|
| **ContentDelta** | AI 生成文本片段 | 追加到 Assistant 消息内容 |
| **ThinkingDelta** | AI 显示思考过程 | 追加到 reasoning 字段 |
| **ToolUseStart** | AI 开始调用工具 | 记录工具调用信息 |
| **ToolUseStop** | AI 完成工具参数填写 | 标记工具调用完成 |
| **Error** | 流发生错误 | 中断处理 |
| **Complete** | AI 完成回复 | 更新 finish_reason，追踪 token 使用量 |

### 4.2 实时持久化

每条消息增量都会**实时写入数据库**：

- 用户输入后立即创建 User 消息
- 收到第一个内容增量时创建 Assistant 消息（空内容）
- 每次收到 ContentDelta 都更新数据库
- 工具调用开始和结束也实时更新

**设计意图**

- 即使程序崩溃，对话历史也不会丢失
- TUI 可以实时展示增量内容
- 支持断线重连（虽然当前没实现）

## 5. 并发控制与取消机制

### 5.1 Busy 检测

每个会话同一时间只能处理一个请求：

```
// 使用 sync.Map 存储活跃请求
activeRequests map[sessionID] -> context.CancelFunc

// 检查是否 busy
if IsSessionBusy(sessionID) {
    return ErrSessionBusy
}
```

**设计意图**

- 防止用户连续发送消息导致并发混乱
- 保证消息顺序性

### 5.2 取消机制

每个请求都绑定一个独立的 context：

```
genCtx, cancel := context.WithCancel(ctx)
activeRequests.Store(sessionID, cancel)

// 用户按 Ctrl+C 或 Ctrl+X
Cancel(sessionID) {
    cancel()  // 触发 context 取消
}
```

**取消的级联效应**

1. LLM 流式请求中断
2. 正在执行的工具（bash）收到取消信号
3. 未开始的工具调用标记为"已取消"
4. 消息标记为 `FinishReasonCanceled`

### 5.3 并发安全

- `activeRequests` 使用 `sync.Map`（并发安全）
- 消息更新通过数据库保证一致性
- 事件发布通过 channel（无锁通信）

## 6. 工具调用执行

### 6.1 执行流程

当 AI 返回 ToolUse 时，引擎按以下步骤执行：

1. **解析工具调用**：从 AI 响应中提取工具名和参数
2. **查找工具**：在可用工具列表中匹配
3. **权限检查**
   - 如果是敏感工具（write、bash、edit），询问用户确认
   - 用户可选择"允许一次"或"允许本次会话"
4. **并发执行**：多个独立的工具调用并行执行（通过 goroutine）
5. **结果收集**：所有工具执行完成后，合并为一条 Tool 角色消息
6. **追加历史**：将工具结果消息加入对话历史
7. **继续循环**：再次调用 LLM，让 AI 基于工具结果继续思考

### 6.2 错误处理

工具执行可能失败，引擎设计了完善的错误传递：

- **工具不存在**：返回错误，让 AI 知道调用失败
- **权限拒绝**：标记为错误，AI 可以尝试其他方式
- **执行超时**：bash 命令可以设置超时时间
- **执行异常**：错误信息返回给 AI，AI 可以决定重试或报错

## 7. 辅助功能设计

### 7.1 异步标题生成

当用户在新会话发送第一条消息时，引擎会**异步**启动标题生成：

```
if len(msgs) == 0 {
    go func() {
        generateTitle(sessionID, content)
    }()
}
```

**设计意图**

- 不阻塞主对话流程
- 用轻量级模型（专门配置 Title Agent）生成简短标题
- 生成后更新会话标题，TUI 实时展示

### 7.2 Token 使用追踪

每次对话完成后，引擎会计算并记录：

- Prompt Tokens（输入 token 数）
- Completion Tokens（输出 token 数）
- Cache Read/Create Tokens（缓存命中/创建）
- Cost（根据模型单价计算成本）

**用途**

- 显示给用户看
- 触发自动摘要（当接近上下文上限时）
- 成本分析

---

## 8. 设计亮点总结

| 设计点 | 说明 |
|--------|------|
| **Tool Use Loop** | AI 不再是单轮问答，而是能反复调用工具直到任务完成 |
| **流式事件驱动** | 实时响应，用户体验接近 ChatGPT |
| **实时持久化** | 每条消息增量都写入数据库，防崩溃丢失 |
| **智能截断** | 通过 SummaryMessageID 实现历史压缩，节省 token |
| **并发控制** | sync.Map + context 实现请求管理和取消 |
| **权限中间件** | 工具执行前插入权限检查，保证安全 |
| **异步标题** | 后台 goroutine 生成标题，不阻塞对话 |

---

## 9. 一句话概括

OpenCode 的 Agent 引擎 = **"一个能自主调用工具、持续推理直到任务完成、支持流式输出和随时取消的异步对话处理器"**。

它把 AI 从"问答机器人"变成了"能动手干活的助手"。

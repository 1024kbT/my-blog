# OpenCode 技术架构文档

> 本文档从架构视角介绍 OpenCode 项目的设计思路、模块划分和数据流转，不涉及具体代码实现细节。

---

## 1. 项目概述

OpenCode 是一款面向开发者的终端 AI 助手，它将大型语言模型（LLM）的能力集成到命令行环境中。用户可以通过交互式终端界面与 AI 进行对话，同时让 AI 具备操作本地文件、执行命令、获取代码智能提示等能力。

**核心定位**：终端内的 AI 编程助手，强调与开发工作流的深度集成。

---

## 2. 整体架构

├─────────────────────────────────────────────────────────────┤
│                      应用编排层 (App)                         │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────────┐   │
│  │ 会话管理  │ │ 消息管理  │ │ Agent    │ │   LSP 客户端 │   │

## 3. 各层职责

### 3.1 用户交互层 (TUI)

负责所有终端界面渲染和用户输入处理。

- **页面系统**：聊天主页面、日志查看页面
- **对话框系统**：会话切换、模型选择、权限确认、命令面板、文件选择等
- **编辑器组件**：内置文本编辑器，支持 Vim 风格操作
- **主题系统**：支持多款配色主题，可动态切换
- **布局系统**：分栏布局、悬浮层、容器管理

**设计特点**：基于事件驱动更新界面，与业务逻辑解耦。

### 3.2 应用编排层 (App)

作为系统的协调中心，负责初始化和连接各个服务模块。

- **会话服务**：管理对话生命周期，支持父子会话关系
- **消息服务**：处理消息的存储、查询和序列化
- **Agent 服务**：协调 AI 对话流程，管理多轮交互
- **权限服务**：控制 AI 工具调用的授权策略
- **LSP 客户端**：管理多语言服务器连接

**设计特点**：通过服务接口抽象各模块，便于测试和替换实现。

### 3.3 业务逻辑层 (LLM)

核心 AI 能力层，封装与语言模型交互的全部逻辑。

- **提供商抽象**：统一接口封装多家 AI 服务商（OpenAI、Anthropic、Google、Azure 等）
- **提示词管理**：按场景组织提示词模板（编码、任务、摘要、标题生成）
- **工具系统**：为 AI 提供操作环境的能力集
  - 文件操作（读、写、编辑、补丁）
  - 命令执行（Shell 命令）
  - 代码搜索（文件查找、内容检索）
  - 外部数据获取（URL 抓取）
- **MCP 扩展**：通过 Model Context Protocol 接入外部工具生态

**设计特点**：提供商可插拔，工具可扩展，提示词与逻辑分离。

### 3.4 基础设施层

为上层提供通用能力支撑。

- **配置管理**：支持多层级配置（全局、项目本地），环境变量自动映射
- **数据库**：本地 SQLite 持久化存储会话、消息、文件变更历史
- **日志系统**：分级日志记录，支持持久化存储和实时查看
- **事件总线**：发布-订阅模式的内部通信机制，解耦模块间依赖

## 4. 核心数据流

### 4.1 交互模式数据流

```
用户输入 → TUI 捕获 → App 分发 → Agent 处理 → 调用 LLM API
                                              ↓
用户查看 ← TUI 更新 ← 事件总线 ← 结果返回 ← 工具调用（可选）
```

**说明**：
1. 用户在 TUI 输入消息
2. 消息经 App 层传递到 Agent
3. Agent 构造请求发送给 LLM 提供商
4. LLM 可能要求调用工具（如读文件、执行命令）
5. Agent 执行工具并将结果返回给 LLM
6. 最终回复通过事件总线广播到 TUI 展示

### 4.2 非交互模式数据流

```
命令行参数 → App 初始化 → Agent 单次执行 → 输出到 stdout → 进程退出
```

**说明**：支持脚本化使用，适合 CI/CD 或自动化场景。

### 4.3 文件变更追踪流

```
工具修改文件 → 记录变更 → 数据库存储 → TUI 展示 diff → 用户确认/回滚
```

## 5. 关键架构决策

### 5.1 终端优先

选择终端作为核心交互场景，而非 Web 或桌面应用：
- 与开发工作流无缝融合
- 低资源占用
- 跨平台兼容性好
- 适合远程服务器环境

### 5.2 本地优先的数据存储

使用本地 SQLite 而非云服务：
- 对话数据完全本地留存
- 支持离线查阅历史
- 符合开发者对隐私的偏好
- 零外部依赖

### 5.3 多模型策略

同时支持多家 AI 提供商：
- 用户可根据场景选择最优模型
- 避免供应商锁定
- 支持本地部署模型（通过 OpenAI 兼容接口）
- 不同任务可配置不同模型（如用轻量模型生成标题）

### 5.4 工具化 AI

将 AI 能力通过工具集暴露给环境：
- AI 不只是聊天，而是能实际操作代码库
- 细粒度权限控制（每次敏感操作可要求确认）
- 操作可追溯、可回滚

### 5.5 会话压缩机制

长对话自动摘要：
- 监控 token 使用量
- 接近上限时自动触发摘要
- 生成新会话继承上下文
- 防止因超出上下文窗口导致中断

## 6. 扩展机制

### 6.1 MCP (Model Context Protocol)

标准化的外部工具接入协议：
- 支持 stdio 和 SSE 两种通信方式
- 自动发现可用工具
- 与内置工具统一的权限管理

### 6.2 LSP (Language Server Protocol)

代码智能能力集成：
- 支持多语言服务器并行连接
- 文件变更自动同步到语言服务器
- AI 可通过诊断工具获取代码错误信息

### 6.3 自定义命令

用户可创建预定义提示词模板：
- Markdown 文件定义命令内容
- 支持命名参数占位符
- 支持用户级和项目级命令

---

## 7. 部署与运行模式

### 7.1 交互模式

启动 TUI，提供完整的聊天和编辑体验：
- 适合日常开发辅助
- 支持快捷键操作
- 实时展示 AI 思考过程

### 7.2 非交互模式

命令行直接传入提示词，输出结果后退出：
- 适合脚本和自动化
- 支持 JSON 格式输出
- 静默模式（无动画）

---

## 8. 架构演进方向

基于当前架构，可预见的演进方向包括：

- **更丰富的工具集**：持续扩展 AI 可操作的环境能力
- **更智能的上下文管理**：自动识别相关代码文件作为上下文
- **多 Agent 协作**：不同角色 Agent 协同处理复杂任务
- **更深的 IDE 集成**：在保持终端核心的同时，提供编辑器插件
- **知识库集成**：支持 RAG（检索增强生成）接入项目文档

---

## 9. 总结

OpenCode 的架构围绕"**终端内的 AI 开发伙伴**"这一核心定位展开，通过清晰的分层设计将界面、业务逻辑和基础设施解耦。其关键设计亮点在于：

1. **TUI + Agent + Tools** 三位一体的交互模型
2. **多提供商抽象** 带来的灵活性
3. **本地优先** 的隐私和离线友好设计
4. **事件驱动** 的模块间通信机制
5. **可扩展的工具生态**（内置工具 + MCP + LSP）

这种架构使得 OpenCode 既能提供流畅的交互体验，又能保持足够的扩展性以适应快速发展的 AI 技术生态。

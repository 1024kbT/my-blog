# 构建终端 AI 助手的核心功能清单

基于 OpenCode 项目架构，梳理构建类似软件需要关注的核心模块和实现要点。

---

## 1. 整体架构分层

```
┌────────────────────────────────────────────┐
│  CLI 入口层                                  │
│  - 命令行参数解析                             │
│  - 交互模式 / 非交互模式 路由                  │
├────────────────────────────────────────────┤
│  应用编排层 (App)                            │
│  - 服务初始化与生命周期管理                    │
│  - 会话、消息、Agent、权限等模块协调           │
├────────────────────────────────────────────┤
│  核心能力层                                  │
│  - LLM 多提供商集成                          │
│  - Agent 多轮对话引擎                        │
│  - 工具集 (文件/命令/搜索)                   │
│  - 提示词模板系统                            │
│  - 记忆机制 (项目记忆 + 会话摘要)             │
├────────────────────────────────────────────┤
│  基础设施层                                  │
│  - 配置管理                                  │
│  - 本地数据持久化 (SQLite)                   │
│  - 事件总线 (Pub/Sub)                        │
│  - 日志系统                                  │
└────────────────────────────────────────────┘
```

## 2. CLI 入口层

### 需要实现的功能
- **交互模式**：启动 TUI（终端用户界面）
- **非交互模式**：命令行直接传入 prompt，输出结果后退出（适合脚本化使用）
- **参数解析**：debug、工作目录、输出格式等

### 核心关注点
- 模式路由：根据是否有 `-p` 参数决定走交互还是非交互流程
- 初始化顺序：配置加载 → 数据库连接 → Agent 初始化 → TUI 启动
- 优雅退出：确保资源释放、连接关闭

---

## 3. 配置管理

### 需要实现的功能
- 多层级配置：全局配置 (`~/.config`) + 本地配置 (项目目录)
- 环境变量自动映射：API Key、工作目录等
- 提供商自动发现和默认模型设置

### 核心关注点
- **配置优先级**：命令行参数 > 环境变量 > 本地配置 > 全局配置
- **Provider 配置**：每个 LLM 提供商的 API Key、Endpoint、模型列表
- **Agent 配置**：不同角色使用的模型、max_tokens 等
- **上下文路径**：定义哪些文件自动注入提示词（如 OpenCode.md）

---

## 4. LLM 多提供商集成

### 需要实现的功能
- 统一接口封装多家 LLM API（OpenAI、Anthropic、Gemini、本地模型等）
- 流式响应 (SSE/Streaming)
- Token 使用量追踪和成本计算
- 模型能力检测（是否支持附件、Reasoning 等）

### 核心关注点
- **Provider 抽象接口**：
  ```go
  type Provider interface {
      SendMessages(msgs []Message, tools []Tool) (Response, error)
      StreamResponse(msgs []Message, tools []Tool) <-chan Event
      Model() Model
  }
  ```
- **事件流设计**：区分 ContentDelta、ToolUseStart、ToolUseStop、ThinkingDelta、Error、Complete 等事件类型
- **消息格式转换**：统一内部 Message 结构，转换为各提供商的 API 格式
- **工具描述格式**：不同提供商对 function calling/tool use 的 schema 要求不同（OpenAI 的 functions 格式 vs Anthropic 的 tools 格式）

---

## 5. Agent 多轮对话引擎

### 需要实现的功能
- 用户消息接收和发送
- 流式响应实时展示
- **Tool Use Loop**：AI 调用工具 → 执行工具 → 返回结果 → AI 继续思考 → ... 直到完成
- 对话历史管理
- 取消和中断处理

### 核心关注点

#### 5.1 对话流程
```
用户输入
  ↓
构造消息历史（包含之前的对话 + 系统提示词）
  ↓
调用 LLM StreamResponse
  ↓
实时接收事件流（内容增量、工具调用请求）
  ↓
如果 AI 请求调用工具：
  - 解析工具调用参数
  - 执行工具（可能涉及权限确认）
  - 构造工具结果消息
  - 将工具结果追加到历史
  - 再次调用 LLM（继续对话循环）
  ↓
如果 AI 完成回答：保存最终消息，结束
```

#### 5.2 关键设计
- **消息历史截断**：当会话存在 SummaryMessageID 时，只加载摘要之后的消息
- **并发安全**：同一时刻一个会话只能有一个请求在处理
- **上下文传递**：将 sessionID、messageID 注入 context，供工具使用
- **附件处理**：支持图片等二进制内容（如果模型支持）

---

## 6. 提示词模板系统

### 需要实现的功能
- 按 Agent 角色定义不同的系统提示词（Coder、Task、Title、Summarizer）
- 根据 LLM 提供商微调提示词风格
- 动态注入环境信息（工作目录、平台、日期等）
- 加载项目级上下文文件（OpenCode.md 等）

### 核心关注点

#### 6.1 三层组装
```
[基础角色提示词]      ← 根据 Agent 类型和 Provider 选择
        +
[环境信息]            ← 动态生成：工作目录、Git状态、平台、日期、项目文件列表
        +
[项目上下文]          ← 从 contextPaths 读取的文件内容
        ↓
[最终系统提示词]      ← 发送给 LLM
```

#### 6.2 角色分离的意义
- **Coder**：详细的编码规范、工具使用策略、代码风格要求
- **Task**：简洁直接，适合快速问答
- **Title**：一句话概括能力
- **Summarizer**：提取关键信息保留上下文

#### 6.3 项目上下文加载
- 使用 `sync.Once` 缓存，避免重复读取
- goroutine 并发读取多个文件
- 支持目录递归读取和文件去重

---

## 7. 记忆机制

### 7.1 项目级记忆（OpenCode.md）

**目的**：解决跨会话共享项目上下文的问题。

**实现方式**：
- 在项目根目录放置 `OpenCode.md`（Markdown 格式）
- 系统启动时自动读取并注入到 Coder/Task Agent 的系统提示词
- 内容可包括：常用命令、代码风格、项目结构说明
- AI 在使用过程中可以主动建议更新此文件

**核心价值**：
- 跨会话持久化项目知识
- 可纳入版本控制，团队协作共享
- 兼容 Cursor/Copilot 等现有生态（同时读取 `.cursorrules`、`.github/copilot-instructions.md`）

### 7.2 会话级记忆（会话摘要）

**目的**：解决长对话超出 LLM 上下文窗口的问题。

**实现方式**：
- 当对话 token 接近上限（如 95%）时，触发自动摘要
- 或由用户手动触发摘要命令
- 调用 Summarizer Agent 生成对话摘要
- 将摘要保存为当前会话的一条 Assistant 消息
- 设置 `Session.SummaryMessageID` 指向该消息

**消息加载策略**：
```go
if session.SummaryMessageID != "" {
    // 找到摘要消息索引，只保留从摘要开始的消息
    msgs = msgs[summaryIndex:]
}
```

**核心价值**：
- 用摘要替代原始历史，节省 token
- 保留关键上下文（做了什么、正在做什么、修改了哪些文件）
- 支持手动触发和自动触发两种模式

## 8. 工具集 (Tool System)

### 需要实现的功能
为 AI 提供操作本地环境的能力，使其不只是聊天，而是能实际完成任务。

### 核心工具类型

#### 8.1 文件操作工具
- **view**：查看文件内容（支持 offset/limit 分页）
- **write**：写入/创建文件
- **edit**：编辑文件内容（字符串替换）
- **patch**：应用 diff 补丁
- **ls**：列出目录内容
- **glob**：按模式查找文件
- **grep**：搜索文件内容

#### 8.2 命令执行工具
- **bash**：执行 Shell 命令（可设置超时）
- 需要配置使用的 shell（bash/zsh 等）

#### 8.3 外部数据工具
- **fetch**：抓取 URL 内容（转换为 markdown）
- **sourcegraph**：跨仓库代码搜索

#### 8.4 代码智能工具
- **diagnostics**：获取 LSP 诊断信息（错误、警告）

### 工具架构设计

```go
type Tool interface {
    Info() ToolInfo                    // 工具名称、描述、参数 schema
    Run(ctx context.Context, input ToolCallInput) (ToolResult, error)
}
```

**关键设计点**：
- **Schema 定义**：每个工具需要提供 JSON Schema 描述，供 LLM 理解如何调用
- **权限控制**：敏感操作（bash、write、edit）执行前需要用户确认
- **错误处理**：工具执行失败时返回错误信息，让 AI 决定是否重试或换方案
- **并发执行**：支持在一次 LLM 调用中并行执行多个独立工具

---

## 9. 会话与消息管理

### 9.1 会话管理
- **会话创建**：生成唯一 ID，支持父子会话关系
- **会话列表**：查询所有会话，展示标题、消息数、更新时间
- **会话切换**：在 TUI 中快速切换不同会话
- **会话删除**：清理历史数据

### 9.2 消息管理
- **消息创建**：区分 User、Assistant、Tool 角色
- **消息 Parts**：一条消息可包含多个内容片段（文本、工具调用、工具结果、二进制数据、结束标记）
- **消息序列化**：将复杂结构序列化为 JSON 存储
- **消息更新**：流式响应过程中实时更新消息内容

### 9.3 数据模型
```go
Session {
    ID, ParentSessionID, Title
    MessageCount, PromptTokens, CompletionTokens
    Cost, SummaryMessageID
    CreatedAt, UpdatedAt
}

Message {
    ID, SessionID, Role, Parts
    Model, CreatedAt, UpdatedAt, FinishedAt
}
```

## 10. 权限管理

### 需要实现的功能
- 控制 AI 对敏感工具的访问
- 支持单次允许、会话允许、拒绝三种策略
- 持久化权限决策

### 核心关注点
- **权限级别**：
  - `allow`：允许本次执行
  - `allow_session`：允许当前会话内不再询问
  - `deny`：拒绝执行
- **工具分类**：
  - 只读工具（view、ls、glob）通常无需确认
  - 写操作工具（write、edit、bash）需要确认
- **非交互模式**：自动批准所有权限（适合脚本化使用）

---

## 11. 事件总线 (Pub/Sub)

### 需要实现的功能
- 模块间解耦通信
- 支持订阅/发布模式
- 类型安全的事件传递

### 核心关注点
- **Broker 实现**：泛型设计，支持不同类型事件
- **事件类型**：Session 事件、Message 事件、Agent 事件、Permission 事件、Log 事件
- **生命周期**：与 context 绑定，支持取消和清理
- **背压处理**：慢消费者超时丢弃，避免内存泄漏

---

## 12. 本地数据持久化

### 需要实现的功能
- 会话和消息本地存储
- 文件变更历史追踪
- 数据迁移（Schema 升级）

### 核心关注点
- **数据库选型**：SQLite（零配置、单文件、跨平台）
- **WAL 模式**：提升并发性能
- **迁移管理**：使用 goose 等迁移工具管理 schema 变更
- **代码生成**：使用 sqlc 从 SQL 生成类型安全的数据库操作代码

---

## 13. 日志系统

### 需要实现的功能
- 分级日志（Debug/Info/Warn/Error）
- 持久化日志文件
- 实时日志查看

### 核心关注点
- **双通道输出**：控制台（TUI）+ 文件
- **结构化日志**：支持 key-value 字段
- **TUI 集成**：日志通过事件总线推送到 Logs 页面
- **工具结果记录**：可选保存工具调用结果到 JSON 文件（Debug 模式）

## 14. 非核心功能（简要说明）

以下功能对核心体验影响较小，初期可简化实现：

| 功能 | 说明 | 优先级 |
|------|------|--------|
| **TUI 主题** | 多套配色方案切换 | 低 |
| **页面路由** | Chat 页、Logs 页切换 | 低 |
| **对话框** | Help、Session 切换、Model 选择等弹窗 | 低 |
| **图片渲染** | 终端内显示图片 | 低 |
| **LSP 集成** | 语言服务器协议，提供代码诊断 | 中 |
| **MCP 扩展** | 通过外部协议扩展工具集 | 中 |
| **自定义命令** | 用户预定义快捷指令 | 低 |
| **多语言支持** | i18n 国际化 | 低 |

## 16. 最小可行产品 (MVP) 范围

如果从头构建，建议按以下顺序实现：

### Phase 1：基础对话
- CLI 入口（交互模式）
- 配置管理（API Key、模型选择）
- LLM Provider 接口（至少实现一家）
- 简单消息历史（内存存储）
- 基础 TUI（输入框 + 消息展示）

### Phase 2：工具能力
- 工具抽象接口
- 文件查看工具（view）
- 命令执行工具（bash）
- 权限确认机制
- Tool Use Loop（多轮工具调用）

### Phase 3：持久化与记忆
- SQLite 数据库（会话、消息存储）
- OpenCode.md 项目记忆
- 会话摘要（手动触发）

### Phase 4：进阶功能
- 多提供商支持
- 流式响应
- 会话管理（切换、删除）
- 非交互模式
- 自动摘要

## 总结

构建终端 AI 助手的**核心挑战**在于：

1. **Tool Use Loop 的可靠性**：AI 调用工具、执行、返回结果、继续对话的循环要稳定
2. **提示词工程**：不同角色、不同提供商的提示词调优
3. **上下文管理**：在有限的 token 窗口内，合理组织系统提示词、项目记忆、对话历史
4. **权限安全**：AI 能执行命令和修改文件，必须有完善的权限控制
5. **流式体验**：响应要实时展示，不能等全部生成完再显示

抓住这五个核心，就能构建出一个可用的终端 AI 助手。
